# Use one deterministic Translator semantic-mapping seam

The Translator previously mapped SysForm XML into a target-shaped legacy object, removed that target knowledge while building a Source Draft, and then mapped the same semantics again while producing a DSL Draft. Source Intake will instead emit source facts directly, and Source Draft to DSL Draft will be the sole deterministic target-semantic mapping seam; one orchestrator may delegate to focused form, layout, form-rule, script, and workflow mappers without collapsing them into a single function or changing the DSL-only boundary into the Executor.

**Status**: accepted

## Consequences

- Source Intake may parse, match, reconcile, and normalize source facts, but it must not perform authoritative NewOA/MK target selection or emit executable target semantics.
- Source Intake preserves raw designer and metadata evidence while deriving target-neutral Source Kinds. Source-side matching compares source characteristics, never target components such as `xform-address`.
- When a derived source entity such as a description has no natural ID, Source Intake may assign a deterministic, collision-checked Canonical Source Identity that satisfies the unchanged Source Draft/DSL identity constraints. Natural source IDs are never rewritten; the raw marker and evidence remain available, and the mapper preserves the canonical identity unchanged.
- Source Intake may derive canonical Source Semantic Facts when they remain meaningful without NewOA/MK, including visibility, editability, requiredness, source conditions and events, and workflow graph relationships. Each normalized fact retains its raw source mode, attributes, or evidence.
- Target components, target APIs, target workflow node types, MK JavaScript, target layout structures, and executable target status are not Source Semantic Facts. Target API names may appear in Source Draft only as non-authoritative Mapping Hints; authoritative target selection belongs after the Source Draft boundary.
- Existing function-audit `targetApis` and `mkFunction` remain compatibility Mapping Hints. `mkFunction` is inert catalog or external-whitelist text that may feed `functionMappings`, review scaffolding, and diagnostics, but cannot be copied directly into an action function or treated as trusted code; the mapper revalidates referenced target APIs. Existing source-rule `translationStatus: "executable"` means only that the deterministic source recognizer understood the source pattern; the mapper must derive and validate the DSL execution status independently.
- Mapping a Source Kind to a DSL type, target component, props, or executable target behavior belongs exclusively to the Translator semantic-mapping seam.
- “One seam” refers to deterministic Source Draft to DSL Draft mapping. The existing AI Agent Review remains a separate post-mapping review/repair stage with restricted, evidence-backed patches, local DSL validation, and trust validation; redesigning that contract is outside this refactor.
- Before calling an AI provider, Agent Review locally blocks any Unrepairable Mapping Error that cannot be resolved by its existing legal replacement patches. Warnings, review candidates, and repairable existing-field or existing-action targets still enter the unchanged review contract; full DSL and trust validation remains after patch application.
- SysForm and LBPM Source Intake adapters return internal source-side fact fragments, including only explicitly allowed Mapping Hints. `cleanSourceFile` remains the sole public entry point that validates supported file pairing and template identity, assembles those fragments, and owns the versioned Source Draft envelope.
- This refactor preserves the `2.0-source-draft` schema, serialized shape, and deterministic values for supported Route cases. Source Kind is the composite of existing `sourceType` and an explicit allowlist of discriminator fields in `sourceProps`; the mapper cannot use arbitrary raw attributes as hidden mapping inputs. Introducing a new field or version requires separate Route-fixture evidence.
- Existing Source Draft, DSL Draft, and final DSL outputs remain semantically equivalent for every supported Route case.
- `draftSourceDraft(sourceDraft, options)` remains the stable public mapping facade. It directly delegates to focused internal form, layout, form-rule, script, and workflow mappers; those family mappers are not re-exported as additional public seams.
- The mapping implementation uses explicit function composition rather than a registry, plugin system, or dynamic family dispatch. Source Intake cannot depend on component selection, target snippets, validation policy, Executor, or persistence modules. The sole catalog exception is `function-whitelist.js`, which may read `FUNCTION_CATALOG` and `JS_METHOD_CATALOG` only for source-call audit and non-authoritative Mapping Hints; target-mapping modules may use Target Catalogs.
- The Translator form mapper exclusively owns Source Kind to DSL type, `componentId`, and props-selection policy. Target Catalogs validate the chosen target capabilities but contain no source-specific selection rules; duplicate `mkForFieldType`-style intake mapping is removed.
- Executor Native Projection remains a separate downstream responsibility that maps Trusted DSL into NewOA persistence structures; it is not another source-to-DSL mapping seam.
- Modules that currently mix source extraction with target generation are split along the Source Draft boundary; in particular, JSP source-fact extraction remains in Source Intake while MK action generation belongs to the script mapper.
- Form Source Kind mapping is exhaustive and fail-closed. An unsupported or ambiguous form Source Kind never falls back to `text` or `xform-input`; the Source Draft retains its evidence, the DSL Draft records a source-referenced mapping error without fabricating a target field, and Agent Review and trust promotion remain blocked.
- Workflow retains its existing fail-closed diagnostic behavior: an Unmapped Workflow Node may use `review/manualTask` with `pending_review` in DSL Draft solely to preserve graph shape, and trust validation prevents it from entering the Executor. No tracked workflow fixture currently exercises this fallback, so changing or expanding it is outside this refactor.
- A malformed Source Draft that violates artifact invariants, such as a missing identity or `sourceRef`, is rejected as invalid input rather than represented as a reviewable mapping gap.
- Adapter tests assert source facts; DSL-mapping and Route-case tests assert target semantics and end-to-end equivalence.
- Target-shaped legacy intermediate objects and compatibility seams are removed rather than retained alongside the new path.

## Verification

- Tracked minimal form-only and paired XML fixtures provide checked-in canonical Source Draft and DSL Draft goldens captured before the refactor. Object-key order is irrelevant, while arrays, identities, `sourceRef` values, diagnostics, Mapping Hints, and all other deterministic values remain exactly equal; only explicitly enumerated volatile metadata may be normalized.
- Trusted DSL and Executor verification continue through fake-review Route semantics, dry-run and execution outcomes, operation sequences, and readback invariants. Runtime IDs and complete NewOA payloads are not compared as whole-object goldens.
- Source Intake tests reject authoritative target leakage, including `componentId`, legacy `mk`, and `@elem/*` identifiers. Target API names and code-shaped examples are allowed only in the exact `functionAudit` Mapping Hint paths for `targetApis` and `mkFunction`; all other MK code is rejected.
- Table-driven form-mapper tests cover every supported form Source Kind and the target type, component, and props choice. An unknown form Source Kind proves structured fail-closed behavior with no text-input fallback.
- Static dependency tests keep Source Intake independent from Executor, target persistence, component-selection, target-snippet, and validation-policy modules, while allowlisting only the named function-audit catalog edge.
- A fake provider that fails if called proves Unrepairable Mapping Errors are blocked locally before Agent Review transport, while repairable candidates still exercise the existing provider and patch path.
- Deletion checks prove that `sourceDraftFromLegacyDsl`, target-shaped XML adapter output, and intake-side `mkForFieldType` mapping no longer have production references.
- The complete default verification path is offline and uses only tracked fixtures and fake clients/providers; it does not access AI, NewOA, browser automation, or untracked local corpora.
- The default test command explicitly disables local-corpus tests even when the parent shell exports `MK_LOCAL_CORPUS_TESTS=1`, and a tracked-fixture inventory guard proves every default fixture dependency is committed.

## Delivery scope

The change is delivered as one complete deterministic Translator boundary while implementation proceeds through test-green slices: add characterization and guard tests; make SysForm and LBPM intake return source-fact fragments assembled behind `cleanSourceFile`; consolidate form target mapping and remove legacy `mk` output; separate JSP source-fact extraction from MK action generation and establish the internal family mappers; adapt LBPM without changing DAG semantics; delete the legacy intermediate path and duplicate helpers; then run the complete tracked-fixture offline suite.

The final change contains no old/new dual runtime path and does not change the Source Draft or DSL schemas, redesign the Agent Review patch contract, modify Executor Native Projection, add a source format, or expand supported capability or product scope.
